## Test Scenarios (Given-When-Then Format)

### TS-1: 한국어 텍스트 렌더링 (REQ-1)

**Given**: 사용자가 브라우저에서 대시보드에 접속함
**When**: 페이지가 로드됨
**Then**: 모든 텍스트가 깨짐 없이 올바른 한국어 폰트로 표시됨

**Acceptance Criteria**:
- [ ] 차트 제목, 축 라벨, 범례가 한국어로 표시됨
- [ ] 뉴스 기사 제목의 한글 문자가 올바르게 렌더링됨
- [ ] 특수 문자(물결표 ~, 마이너스 기호 -)가 깨지지 않음

**Test Data**:
- news_daily.csv에 한국어 기사 제목 포함
- 차트 라벨: "날짜", "감성 점수", "금리(%)"

---

### TS-2: 대시보드 초기 로드 (REQ-4)

**Given**: 백엔드 파이프라인이 실행되어 최신 CSV 파일이 존재함
**When**: 사용자가 루트 URL("/")에 접속함
**Then**: 대시보드에 4개 주요 차트가 렌더링됨

**Acceptance Criteria**:
- [ ] 페이지 로드 시간 3초 이내
- [ ] 뉴스 감성 시계열 차트 표시됨
- [ ] 금리 시계열 차트 표시됨
- [ ] 이벤트 스터디 차트 표시됨
- [ ] 통계 카드 4개 표시됨
- [ ] 뉴스 기사 리스트 표시됨

**Performance Baseline**:
- Initial Page Load: < 3s (Fast 3G)
- Time to Interactive: < 5s
- First Contentful Paint: < 1.5s

---

### TS-3: 호버 툴팁 표시 (REQ-5)

**Given**: 대시보드가 로드되고 차트가 표시됨
**When**: 사용자가 금리 시계열 차트의 데이터 포인트에 마우스를 호버함
**Then**: 해당 날짜의 금리 값과 뉴스 기사 수가 툴팁으로 표시됨

**Acceptance Criteria**:
- [ ] 툴팁이 마우스 위치 근처에 표시됨
- [ ] 툴팁에 날짜(YYYY-MM-DD), 금리 값, 기사 수 포함됨
- [ ] 툴팁 배경색이 불투명하여 텍스트가 가려지지 않음
- [ ] 마우스를 이동하면 툴팁이 부드럽게 따라옴

**Test Cases**:
1. 금리 인상 일자 호버 → 툴팁에 "금리 인상" 라벨 표시
2. 데이터 없는 날짜 호버 → "데이터 없음" 메시지 표시

---

### TS-4: 이벤트 상세 패널 (REQ-6)

**Given**: 대시보드가 로드되고 금리 변화 이벤트가 차트에 표시됨
**When**: 사용자가 특정 이벤트 마커를 클릭함
**Then**: 해당 이벤트의 상세 정보가 사이드 패널에 표시됨

**Acceptance Criteria**:
- [ ] 사이드 패널이 우측에서 슬라이드하여 나타남
- [ ] 이벤트 날짜, 유형(인상/인하/유지), 설명 표시됨
- [ ] 해당 이벤트와 관련된 뉴스 기사 리스트 표시됨
- [ ] "닫기" 버튼 또는 바깥 영역 클릭 시 패널이 닫힘
- [ ] 패널이 열려 있을 때 대시보드 스크롤 가능

**Test Data**:
- 이벤트 유형: "금리 인상", "금리 인하", "금리 유지"
- 관련 뉴스: 이벤트 날짜 ±3일 내 뉴스 기사

---

### TS-5: 데이터 새로고침 기능 (REQ-7)

**Given**: 대시보드가 로드되고 초기 데이터가 표시됨
**When**: 사용자가 "데이터 새로고침" 버튼을 클릭함
**Then**: 백엔드 API에서 최신 CSV 데이터를 가져와 차트가 업데이트됨

**Acceptance Criteria**:
- [ ] 버튼 클릭 시 로딩 스피너가 표시됨
- [ ] API 요청 중에는 다른 버튼들이 비활성화되지 않음
- [ ] 데이터 가져오기 성공 시 "성공적으로 업데이트됨" 토스트 메시지 표시
- [ ] 데이터 가져오기 실패 시 "업데이트 실패" 에러 메시지와 "재시도" 옵션 표시
- [ ] 차트 데이터가 실제로 업데이트됨 (날짜, 값 비교)

**Test Cases**:
1. 정상 응답 (200 OK) → 차트 업데이트, 성공 메시지
2. 서버 에러 (500) → 에러 메시지, 재시도 버튼
3. 네트워크 에러 → 오프라인 메시지, 재시도 버튼

---

### TS-6: 감성 점수 색상 코딩 (REQ-10, REQ-11)

**Given**: 뉴스 감성 시계열 차트가 표시됨
**When**: 차트가 렌더링됨
**Then**: 양수 감성 점수는 빨간색, 음수 감성 점수는 파란색으로 표시됨

**Acceptance Criteria**:
- [ ] 감성 점수 > 0: 빨간색 계열 (#ef4444)
- [ ] 감성 점수 < 0: 파란색 계열 (#3b82f6)
- [ ] 감성 점수 = 0: 중립 색상(회색 #9ca3af)
- [ ] 색상 범례(Legend)에 "매파(빨강)", "비둘기(파랑)" 라벨 표시

**Test Data**:
- news_daily.csv에 stance_mean 범위: -0.8 ~ +0.8

---

### TS-7: 반응형 모바일 레이아웃 (REQ-12)

**Given**: 대시보드가 데스크톱에서 정상적으로 표시됨
**When**: 사용자가 브라우저 창 너비를 768px 미만으로 조절함
**Then**: 모바일 최적화 레이아웃으로 전환됨

**Acceptance Criteria**:
- [ ] 차트가 세로로 쌓이는 단일 열 레이아웃으로 변경됨
- [ ] 통계 카드가 2x2 그리드에서 4x1 단일 열로 변경됨
- [ ] 폰트 크기가 적절히 축소됨 (14px → 12px)
- [ ] 버튼 크기가 터치 가능한 크기로 유지됨 (최소 44px 높이)
- [ ] 햄버거 메뉴 (선택 사항)

**Breakpoints**:
- Mobile: < 640px (1열)
- Tablet: 640px - 1024px (2열)
- Desktop: > 1024px (혼합 레이아웃)

---

### TS-8: 부드러운 페이지 전환 (REQ-16)

**Given**: 대시보드가 로드되고 사용자가 페이지 간 이동함
**When**: 사용자가 차트 간 전환 또는 데이터 필터링을 수행함
**Then**: 페이지 깜빡임(flash) 없이 부드러운 전환 효과가 적용됨

**Acceptance Criteria**:
- [ ] 데이터 리페칭 시 전체 페이지가 깜빡이지 않음
- [ ] 차트 컴포넌트만 교체되고 배경은 유지됨
- [ ] 로딩 상태가 시각적으로 표시됨 (Skeleton, Spinner)
- [ ] 전환 애니메이션 시간 200-300ms

**Visual QC**:
- 수동 테스트: 데이터 새로고침 클릭 → 깜빡임 없는지 확인

---

### TS-9: 로딩 상태 관리 (REQ-17)

**Given**: 대시보드가 로드됨
**When**: 사용자가 데이터를 요청하는 작업을 수행함 (새로고침, 필터링)
**Then**: UI가 응답하지 않는 상태(freeze)가 되지 않고 로딩 상태가 표시됨

**Acceptance Criteria**:
- [ ] 로딩 중에도 사용자가 다른 UI 요소와 상호작용 가능함
- [ ] 로딩 스피너가 적절한 위치에 표시됨 (차트 중앙 또는 상단)
- [ ] 로딩 시간이 5초 초과 시 "로딩 중..." 메시지 표시
- [ ] 로딩 시간이 10초 초과 시 타임아웃 에러 표시

**Performance Test**:
- API 응답 지연: 1초, 3초, 5초 (각각 로딩 상태 확인)

---

### TS-10: 에러 핸들링 (REQ-8)

**Given**: 대시보드가 로드됨
**When**: API 요청이 실패함 (네트워크 에러, 서버 에러)
**Then**: 사용자에게 명확한 에러 메시지가 표시되고 재시도 옵션이 제공됨

**Acceptance Criteria**:
- [ ] 에러 메시지가 사용자 친화적인 언어로 표시됨 (한국어)
- [ ] 에러 유형별로 구체적인 메시지 제공:
  - 네트워크 에러: "인터넷 연결을 확인해주세요"
  - 서버 에러: "서버 오류가 발생했습니다. 잠시 후 다시 시도해주세요"
  - 데이터 없음: "표시할 데이터가 없습니다"
- [ ] "재시도" 버튼이 제공됨
- [ ] 치명적 에러가 아닌 경우 다른 기능 계속 사용 가능

**Test Cases**:
1. 네트워크 차단 (DevTools) → 네트워크 에러 메시지
2. 백엔드 서버 중지 → 서버 에러 메시지
3. 빈 CSV 파일 → 데이터 없음 메시지

## Definition of Done

**"완료" 정의**:

1. **기능 완료**: 모든 필수 요구사항(REQ-1 ~ REQ-12)이 구현되고 테스트 통과
2. **성능 달성**: Lighthouse Performance 점수 90+ (Mobile)
3. **접근성**: WCAG 2.1 Level AA 준수
4. **브라우저 호환성**: Chrome, Safari, Firefox 최신 2버전 작동 확인
5. **코드 품질**: ESLint 경고 0개, TypeScript 컴파일 에러 0개
6. **문서화**: README.md 업데이트 (설치, 실행, 사용법)
7. **배포 준비**: Docker 빌드 성공, docker-compose 실행 가능

**완료되지 않은 경우**:
- 치명적 버그 존재
- 성능 기준 미달
- 접근성 위반
- 주요 브라우저에서 렌더링 오류

## Test Automation Strategy

### 유닛 테스트 (Jest + React Testing Library)

**범위**:
- [ ] 컴포넌트 렌더링 테스트
- [ ] 사용자 인터랙션 테스트 (클릭, 호버)
- [ ] API 클라이언트 함수 테스트 (MSW mocking)

**목표 커버리지**: 80%

### 통합 테스트 (Playwright)

**시나리오**:
1. 사용자가 대시보드에 접속 → 차트 표시 확인
2. 데이터 새로고침 → 차트 업데이트 확인
3. 이벤트 클릭 → 상세 패널 표시 확인
4. 모바일 레이아웃 → 반응형 전환 확인

**실행 환경**: Chrome, Mobile Safari (Emulation)

### E2E 테스트 (선택 사항)

**시나리오**:
1. 백엔드 파이프라인 실행 → CSV 파일 생성
2. 프론트엔드 서버 실행 → 대시보드 로드
3. 데이터 새로고침 → 최신 데이터 확인

## Bug Triage

**심각도 분류**:

- **Critical**: 기능 작동 불가, 데이터 손실
- **High**: 주요 기능 작동 불가, 사용자 경험 심각히 저하
- **Medium**: 일부 기능 작동 불가, 우회 방법 존재
- **Low**: 사소한 UI 문제, 텍스트 오타

**우선순위**:
1. Critical → 즉시 수정 (RCA 포함)
2. High → 2일 이내 수정
3. Medium → 1주 이내 수정
4. Low → 다음 스프린트 포함
