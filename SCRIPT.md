## 슬라이드 1 — 표지 (RATESTANCE)

**[약 1분]**

안녕하세요. 오늘은 제가 개발한 **RateStance**라는 프로젝트를 소개해드리겠습니다.

RateStance는 이름에서 알 수 있듯이, **금리(Rate)**와 **스탠스(Stance)**, 즉 한국은행의 기준금리 결정과 금융 뉴스의 논조 사이에 어떤 관계가 있는지를 자동으로 분석하는 데이터 파이프라인입니다.

오늘 발표는 크게 **문제 정의 → 솔루션과 핵심 기능 → 기술 아키텍처 → 성과 지표 → 향후 로드맵** 순으로 진행하겠습니다.

## 슬라이드 2 — RATESTANCE가 해결하는 문제

**[약 2분]**

먼저 왜 이 프로젝트가 필요했는지부터 시작하겠습니다.

기자나 경제 분석가가 "지금 시장이 금리 인상을 예상하고 있는가, 아니면 인하를 예상하는가?"를 파악하려면 어떻게 해야 할까요? 지금은 **수동으로 뉴스를 읽고 판단**합니다. 시간이 많이 걸리고, 사람마다 해석이 달라집니다.

여기에는 세 가지 핵심 문제가 중첩되어 있습니다.

**첫째, 정책 신호 포착 지연입니다.** 실시간으로 수십, 수백 건의 기사를 검토하는 것은 비효율적입니다. 중요한 정책 신호를 놓칠 수 있습니다.

**둘째, 히스토리 데이터 부재입니다.** 과거 금리 결정 전후에 뉴스가 어떤 논조였는지를 체계적으로 분석한 데이터가 없습니다. "2022년 금리 인상 직전에 미디어 분위기가 어땠나?"라는 질문에 즉시 답하기 어렵습니다.

**셋째, 정량화 부재입니다.** "요즘 분위기가 매파적이다"라는 말은 있지만, 숫자로 표현되지 않습니다. 비교하거나 추적하기가 어렵습니다.

RateStance는 이 세 가지 문제를 **자동화된 파이프라인 하나로** 동시에 해결합니다. 슬라이드의 체인 이미지처럼, 뉴스 수집 → 스탠스 스코어링 → 이벤트 감지 → 이벤트 스터디가 서로 연결되어 하나의 분석 흐름을 만들어냅니다.

## 슬라이드 3 — RATESTANCE의 주요 기능

**[약 3분]**

RateStance는 총 여섯 가지 핵심 기능을 제공합니다. 하나씩 짚어보겠습니다.

**첫 번째, 이중 소스 뉴스 수집입니다.**

뉴스를 두 가지 소스에서 가져옵니다. **2025년 8월 1일 이전**의 과거 데이터는 구글이 운영하는 공개 빅데이터 프로젝트인 **GDELT BigQuery**에서, **그 이후** 현재 뉴스는 **Google News RSS 피드**에서 수집합니다. "한국은행 기준금리", "통화정책" 등 한국어·영어 키워드로 기사를 수집하며, 중복 기사는 자동으로 제거합니다.

**두 번째, 키워드 기반 스탠스 스코어링입니다.**

각 기사에서 매파 키워드와 비둘기파 키워드의 등장 횟수를 세어 스코어를 계산합니다.

`stance_score = 매파 키워드 수 - 비둘기파 키워드 수`

예를 들어 "인상", "긴축", "물가압력" 같은 단어가 많으면 양수(매파), "인하", "완화", "부양" 같은 단어가 많으면 음수(비둘기파)가 됩니다. 키워드 목록은 커스터마이즈 가능합니다.

**세 번째, 6단계 데이터 파이프라인입니다.**

뉴스 수집부터 이벤트 스터디까지 여섯 단계가 완전 자동화되어 있습니다. 버튼 하나로 전체 분석이 실행됩니다.

**네 번째, 이벤트 스터디 분석입니다.**

각 금리 변동 이벤트를 기준으로 전후 14일 창을 만들어서, 뉴스 스탠스가 어떻게 변화하는지를 분석합니다. "금리 인상 2주 전부터 매파 분위기가 고조되는가?" 같은 질문에 데이터로 답할 수 있게 됩니다.

**다섯 번째, 인터랙티브 대시보드입니다.**

Next.js로 구현된 웹 대시보드에서 날짜를 클릭하면 해당 날짜 전후 기사 목록이 나오고, 이벤트 마커를 클릭하면 금리 변동 상세 정보가 표시됩니다.

**여섯 번째, 비동기 Job 관리입니다.**

파이프라인 실행은 백그라운드에서 비동기로 처리되며, 0%부터 100%까지 진행률을 실시간으로 확인할 수 있습니다.

## 슬라이드 4 — 누가 왜 RATESTANCE를 사용하는가?

**[약 1분 30초]**

이 시스템을 누가 쓸 수 있을까요? 크게 세 그룹을 타깃으로 합니다.

**경제학 연구자**입니다. 통화정책 기대와 미디어 보도 간의 관계를 체계적으로 연구할 수 있습니다. 히스토리 데이터와 이벤트 스터디 기능이 학술 분석의 기반을 제공합니다. 수동으로 아카이브를 뒤질 필요 없이, API 한 번이면 수년치 분석 데이터를 얻을 수 있습니다.

**투자사 애널리스트**입니다. 정책 방향 예측을 위한 감성 시그널을 실시간으로 모니터링합니다. 매파/비둘기파 스코어 추이를 통해 다음 금리 결정 방향을 사전에 포착할 수 있습니다.

**중앙은행 연구원과 금융 미디어**입니다. 시장이 정책을 어떻게 인식하고 있는지를 정량적으로 파악할 수 있습니다. 보도 논조와 실제 정책 결정의 정합성을 추적하는 데도 활용할 수 있습니다.

## 슬라이드 5 — 기술 스택 및 성능 현황

**[약 2분]**

기술적인 측면을 보겠습니다.

백엔드는 **Python 3.11+**과 **FastAPI**로 구현했습니다. 데이터 처리에는 **Pandas**, 설정 관리에는 **Pydantic-settings**, 로깅에는 **Loguru**를 사용했습니다. 특히 외부 API 호출 시 **Tenacity** 라이브러리로 재시도 로직을 구현해서, 네트워크 불안정에도 3회까지 지수 백오프로 재시도합니다.

프론트엔드는 **Next.js App Router**, **React 19**, **TypeScript**로 구성되어 있고, 차트는 **Recharts**, UI 컴포넌트는 **Radix UI + Tailwind CSS 4**를 사용했습니다.

성능 지표를 보시면, 목표치와 현재 달성치를 비교했습니다.

코드 커버리지는 목표가 85%인데 현재 **93.24%**로 목표를 초과 달성했습니다. 50개의 테스트가 모두 통과하고 있습니다.

API 응답 시간은 P95 기준 500ms 이내를 목표로 설계했으며, 파이프라인은 6개월치 데이터 기준 5분 이내 처리를 목표로 합니다.

## 슬라이드 6 — 측정 가능한 목표치 (KPI)

**[약 1분 30초]**

RateStance의 성공을 어떻게 측정할까요? 네 가지 핵심 지표로 정의했습니다.

**93%** — 코드 커버리지입니다. 이미 목표 85%를 초과 달성하고 있습니다. 50개 테스트가 지속적으로 코드 품질을 보장합니다.

**99.5%** — API 가용성 목표입니다. 현재 ip9202.site 서버에서 PM2로 프로세스를 관리하며 운영 중입니다.

**24시간** — 데이터 신선도입니다. 최신 뉴스 데이터가 항상 24시간 이내로 유지되는 것을 목표로 합니다.

**R² > 0.3** — 뉴스 스탠스와 금리 결정 사이의 상관계수 목표입니다. 이것이 이 프로젝트의 핵심 연구 가설입니다. 뉴스가 금리를 선행하는가? 현재 이 값을 검증하기 위한 데이터를 축적하고 있습니다.

기술 지표는 이미 목표를 초과 달성하고 있고, 이제 연구 유효성 검증으로 나아가는 단계입니다.

## 슬라이드 7 — 6단계 파이프라인 아키텍처

**[약 2분 30초]**

파이프라인의 내부 구조를 좀 더 자세히 보겠습니다.

**1단계 뉴스 수집 (0~20%):** GDELT BigQuery와 Google News RSS에서 한국어·영어 키워드로 기사를 수집하고 중복을 제거합니다. 과거와 현재 데이터 소스를 날짜 기준으로 자동 분기합니다.

**2~3단계 스코어링 & 집계 (20~60%):** 각 기사 제목과 요약에서 키워드 매칭으로 매파/비둘기파 스코어를 산출하고, 날짜별로 평균 스탠스를 집계합니다. 결과는 `news_scored.csv`와 `news_daily.csv`에 저장됩니다.

**4~5단계 기준금리 수집 & 이벤트 감지 (60~90%):** 한국은행 ECOS API에서 기준금리 시계열을 수집하고, 값의 변화를 감지해 인상/인하/동결 이벤트로 분류합니다. ECOS API는 재시도 로직으로 안정성을 보장합니다.

**6단계 이벤트 스터디 (90~100%):** 각 금리 변동 이벤트를 기준으로 ±14일 창을 생성하여, 그 기간 동안의 뉴스 스탠스 변화 패턴을 분석합니다.

슬라이드 하단의 인용문처럼, 이 분석의 핵심 질문은 이것입니다.

> *"각 금리 변동 이벤트를 기준으로 ±14일 윈도우를 생성하여 뉴스가 정책 결정을 선행하는지, 시장이 인상/인하에 다르게 반응하는지를 정량적으로 검증합니다."*

전체 파이프라인이 완료되면 `data/` 디렉토리에 6개의 CSV 파일과 Matplotlib으로 생성한 300 DPI PNG 차트가 함께 저장됩니다.

## 슬라이드 8 — 향후 로드맵: 다음 단계

**[약 2분]**

마지막으로 앞으로의 계획을 말씀드리겠습니다. 우선순위에 따라 네 가지 방향으로 정리했습니다.

**단기 우선순위 — ML 기반 감성 분류입니다.**

현재 시스템은 키워드 매칭 방식으로, 맥락을 이해하지 못합니다. 예를 들어 "금리 인상 우려가 없다"라는 문장은 '인상'이라는 단어 때문에 매파로 잘못 분류될 수 있습니다. BERT나 GPT 기반 모델을 도입하면 정확도를 15~20% 향상시킬 수 있을 것으로 예상합니다. 기존 키워드 방식과 병행 운영하여 성능을 검증할 계획입니다.

**중기 과제 — 인프라 & 안정성 강화입니다.**

현재 Job 상태가 인메모리로만 관리되어 API 재시작 시 초기화됩니다. PostgreSQL로 영속화하고, 폴링 방식 대신 WebSocket 기반 실시간 스트리밍으로 전환할 계획입니다. 일별 자동 파이프라인 스케줄링도 추가할 예정입니다.

**장기 확장 — 다중 중앙은행 지원입니다.**

현재는 한국은행에 특화되어 있지만, ECB, Fed, BoE로 확장하면 국제 비교 분석이 가능해집니다. Granger 인과성 검정으로 뉴스 선행성을 통계적으로도 검증할 수 있게 됩니다.

**지속 개선 — 배포 & 운영 최적화입니다.**

PM2 프로세스 안정성 모니터링 강화, 병렬 수집 단계 도입으로 파이프라인 속도 개선, API 키 인증 추가 등을 계획하고 있습니다.

## 예상 Q&A

### Q1. 키워드 방식의 정확도는 어느 정도인가요?

정확도를 직접 측정한 수치는 아직 없습니다. 다만 한국은행 통화정책 관련 뉴스에서 매파/비둘기파를 구분하는 핵심 단어들은 비교적 명확하게 구분됩니다. 향후 ML 모델과 결과를 비교해 정확도를 정량화할 계획입니다.

### Q2. GDELT 데이터의 신뢰성은 어떻게 보장하나요?

GDELT는 구글이 지원하는 전 세계 뉴스 아카이브로, 1996년부터의 뉴스 데이터를 포함합니다. 한국 관련 기사 필터링은 V2Locations 필드에서 'KR#' 또는 'Korea'를 포함하는 기사만 추출하는 방식입니다. 물론 GDELT 자체의 수집 완성도에 의존한다는 한계는 있습니다.

### Q3. 왜 2025-08-01을 기준으로 데이터 소스를 분기했나요?

GDELT는 실시간 업데이트 지연이 있고, Google News RSS는 최근 2~4주 데이터만 제공합니다. 두 소스의 장단점을 활용하는 자연스러운 분기점이 그 날짜였습니다.

### Q4. 실제로 뉴스 스탠스와 금리 결정 사이에 상관관계가 있나요?

아직 통계적으로 검증하지 못했습니다. R² > 0.3이라는 수치는 목표치로 설정한 것이고, 현재 데이터를 축적하면서 Granger 인과성 검정 등을 통해 검증해 나갈 계획입니다.

### Q5. 인증이 없는데 보안 우려는 없나요?

현재 제공하는 데이터는 모두 공개 소스(뉴스 RSS, 공개 BigQuery, 한국은행 공개 API)에서 수집한 데이터입니다. 개인정보나 민감 데이터를 다루지 않기 때문에 현 단계에서는 인증 없이 운영 중입니다. 상용 서비스로 확장할 경우 API 키 인증을 추가할 계획입니다.
